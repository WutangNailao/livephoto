# LivePhoto 项目介绍

## 背景

动态照片已经不是一个陌生概念。

苹果有 Live Photo，Android 阵营有 Motion Photo，不同厂商也都有自己的动态照片实现。它们共同的目标都是把一张静态照片与一个短暂的动态片段结合起来，让“照片”不仅仅停留在单帧图像上，而能够保留拍摄前后的瞬间、声音和氛围。

但直到今天，这类能力仍然高度依赖具体平台和厂商生态。

苹果原生 Live Photo 本质上是一张静态图片和一段 MOV 视频的配对资源；Android Motion Photo 更接近单文件思路，但不同品牌、不同实现之间仍然存在格式、元数据和播放行为上的差异。对于开发者来说，动态照片长期缺少一个清晰、稳定、可跨平台的统一容器层。

`livephoto` 项目正是在这个背景下提出的。

它的目标不是复刻苹果内部格式，也不是绑定某个单一生态，而是定义一个开放的、单文件的、可跨端解析和播放的动态照片容器格式，并提供围绕这个格式的工具链实现。

## 这个项目要解决什么问题

当前动态照片生态存在几个非常实际的问题。

### 1. 原生动态照片通常不适合统一存储和传输

以苹果 Live Photo 为代表的方案，通常不是一个单文件，而是：

- 一张静态图
- 一段短视频
- 再加上一套用于关联它们的元数据规则

这种方式在系统相册中工作良好，但一旦进入业务系统，就会出现很多工程上的不便：

- 上传时要处理多文件配对
- 存储时要保证资源不丢失、不脱关联
- 数据库中要维护额外索引关系
- CDN 分发时要同步处理两份资源
- 导出、备份、迁移、复制时容易只带走其中一部分

这意味着“动态照片”在很多系统里无法像普通图片或视频那样，成为一个天然的独立对象。

### 2. 各平台实现碎片化，难以跨端复用

苹果、Google、三星、华为、OPPO、小米等厂商都提供过动态照片能力，但它们背后的组织方式、元数据表达、播放器语义和导入导出机制并不统一。

结果是：

- 前端无法直接用同一套方式处理所有动态照片
- 服务端难以建立统一的处理管线
- iOS、Android、Web、桌面端往往需要各写一套适配逻辑
- 动态照片容易退化成“图片+视频”的业务拼装，而不是正式媒体对象

### 3. 播放能力被原厂框架绑定

很多动态照片方案在系统生态中体验很好，但一旦脱离系统相册或原生框架，就难以直接播放或复用。

例如：

- Web 端无法直接读取和展示原生 Live Photo
- 跨平台 App 不能天然依赖 `PHLivePhotoView`
- 服务端也不能直接把一组平台私有资源当成统一媒体格式处理

这会导致动态照片能力很难被真正做成跨平台产品能力。

### 4. 缺少统一的播放语义

“动态照片”不只是“图片和视频放在一起”。

它还涉及：

- 默认展示哪一帧
- 封面图和视频的关系
- 是否自动播放
- 是否循环
- 是否静音
- 播放结束后回到封面还是停在最后一帧

如果这些语义不在格式层定义清楚，播放器、编辑器、导出器就只能各自猜测，最终导致跨平台体验不一致。

## `livephoto` 的解决方案

`livephoto` 项目给出的方案是：

定义一个自有的单文件二进制容器格式 `.livephoto`，在一个文件中同时封装：

- 封面静态图
- 动态视频
- 元数据 manifest
- 缩略图、EXIF、XMP、哈希、桥接信息等可选资源

同时，围绕这个容器建立：

- Rust 格式实现
- CLI 生成器与检查工具
- Web 播放器
- 面向 Apple Live Photo / Android Motion Photo 的桥接层

换句话说，这个项目的核心不是“再做一种手机相机功能”，而是为动态照片提供一个真正可工程化的基础格式层。

## `.livephoto` 格式的核心价值

### 1. 单文件

`.livephoto` 的第一原则是单文件封装。

动态照片在存储、传输、缓存、归档、复制、上传、下载时，都应被视为一个完整对象，而不是多份资源的脆弱组合。

这会明显简化：

- 对象存储
- 数据库存档
- 文件分发
- CDN 缓存
- API 上传下载
- 聊天消息附件
- 云端同步

### 2. 跨平台

该格式不以苹果系统或 Android 系统为运行前提。

只要实现了解析器，就可以在：

- Web
- iOS
- Android
- macOS
- Windows
- Linux

等不同平台上加载、解析和播放。

### 3. 容器与编码解耦

`.livephoto` 不重新定义图像编码或视频编码，而是把媒体编码视为容器中的 payload。

这意味着它可以承载：

- JPEG / HEIC / HEIF / AVIF / PNG / WebP
- MP4 / MOV / WebM

从而避免把“文件组织方式”和“媒体编码方式”混在一起。

### 4. 明确的播放语义

`.livephoto` 不只是塞入图片和视频，还要求 manifest 定义：

- 时长
- 封面时间点
- 主图和主视频引用关系
- 是否带音频
- 默认播放行为

这使不同实现可以在共同语义下渲染，而不是靠播放器自行猜测。

### 5. 可桥接现有生态

`livephoto` 并不试图否定现有动态照片生态。

相反，它明确把 Apple Live Photo 和 Android Motion Photo 视为桥接对象：

- 可从苹果原生 Live Photo 导入为 `.livephoto`
- 可将 `.livephoto` 导出成苹果可识别的图片+MOV
- 可保留 Android Motion Photo 相关桥接元数据

因此它更像一个中间层和统一层，而不是封闭替代品。

## 这个项目的意义

### 对产品的意义

它让动态照片不再局限于某一个系统相册能力，而可以变成真正的产品级媒体对象。

这意味着：

- 社交应用可以把动态照片作为一等附件处理
- 相册应用可以跨平台管理动态照片
- Web 产品可以直接展示动态照片，而不必退化为 GIF 或普通视频
- 内容平台可以统一处理图片、视频和动态照片

### 对工程的意义

它把“动态照片”从业务拼装提升为正式格式。

这会带来：

- 更清晰的存储模型
- 更稳定的导入导出流程
- 更统一的播放器接口
- 更容易维护的后端处理管线
- 更明确的格式演进策略

### 对生态的意义

目前动态照片更多是一种平台功能，而不是一个开放媒体层。

`livephoto` 项目尝试做的事情，是把这类媒体能力抽象出来，形成一个对开发者友好的、可实现的、可演进的容器层。这对于构建开放工具链、跨端 SDK、编辑器和转码器，都有实际价值。

## 适用场景

这个项目适合的场景包括但不限于：

- 相册应用中的动态照片管理
- 社交应用中的动态照片附件
- 云端媒体归档系统
- 素材管理平台
- Web 端动态照片展示
- 跨平台播放器
- 动态照片导入导出与迁移工具
- 动态照片编辑器

如果一个系统既希望保留“照片”的静态语义，又希望保留短暂动态内容，那么它就很可能适合 `.livephoto`。

## 设计原则

这个项目采用以下设计原则。

### 1. 单文件优先

动态照片应被视为一个对象，而不是多个文件之间的弱关联关系。

### 2. 先定义格式，再扩展工具链

播放器、编辑器、转换器都建立在稳定的容器和元数据规范之上，而不是反过来由某个播放器行为去倒推格式。

### 3. 语义明确

封面图、视频、时间点、播放策略等，都应在格式层有清晰定义。

### 4. 可扩展，但 v1 保持克制

第一版只解决最核心的问题：

- 文件组织
- 主资源引用
- 元数据语义
- 基础桥接

不在一开始就引入过多复杂能力。

### 5. 与单一生态解耦

项目可以桥接 Apple 和 Android 生态，但不会把某个厂商私有结构直接当成格式本体。

## 项目组成

当前项目主要包含：

- 格式规范文档
- Rust 实现库 `livephoto-format`
- 命令行工具 `livephoto-cli`
- 最小 Web 播放器

后续可以继续扩展为：

- iOS SDK
- Android SDK
- 编辑器
- 转码器
- 服务端处理组件

## 与苹果 Live Photo 的关系

这一点需要明确：

`livephoto` 不是苹果原生 Live Photo 格式。

它与苹果 Live Photo 的关系是：

- 可以导入苹果素材
- 可以保留苹果桥接元数据
- 可以导出回苹果生态
- 但不会天然被 Apple Photos 当作原生 Live Photo 识别

也就是说，它的目标不是伪装成苹果内部格式，而是提供一个更通用、更适合工程系统的容器层。

## 项目边界

`livephoto` 当前解决的是“动态照片的容器层问题”，而不是所有媒体问题。

它不试图在 v1 阶段直接解决：

- 自定义视频编码
- DRM
- 系统相册原生集成
- 高级剪辑时间线
- 复杂流式媒体协议

这些能力可以在后续工具链中逐步扩展，但不应影响格式层的清晰性。

## 未来方向

这个项目的长期目标包括：

- 成为动态照片的跨平台容器
- 建立 Apple Live Photo / Android Motion Photo 的统一桥接层
- 提供稳定的解析器、生成器、播放器和编辑器工具链
- 让动态照片在 Web、移动端和服务端拥有一致的处理模型

更进一步地说，`livephoto` 的目标不是只解决“如何播放一张会动的照片”，而是建立一套更开放、更统一的动态图像媒体基础设施。

## 总结

`livephoto` 项目试图解决的是一个长期存在、但一直缺少统一答案的问题：

如何把“静态照片 + 短视频 + 播放语义 + 桥接元数据”组织成一个可跨平台、可存储、可传输、可播放、可演进的正式媒体对象。

`.livephoto` 给出的答案是：

用一个自定义但开放的单文件容器，把动态照片从平台功能提升为格式能力，再围绕这个格式建立工具链。

这不仅能改善存储和传输体验，也能让动态照片真正走出单一生态，成为更通用的跨平台媒体类型。
